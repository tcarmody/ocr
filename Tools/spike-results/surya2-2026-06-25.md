# Spike — Surya 2 (surya-ocr 0.20) on Apple Silicon

Run 2026-06-25. Machine: Apple Silicon (arm64), macOS 26. Goal: a
go/no-go on migrating the layout/OCR sidecar from Surya 1 (0.17, in-
process torch + MPS) to Surya 2 (0.20, single Qwen-VL model served by
llama.cpp). Phase 1 of S-Surya2-Migration.

## Setup

- `surya-ocr==0.20.0` installed cleanly into an isolated uv venv
  (Python 3.12) — **no vllm pulled on macOS** (torch 2.12.1,
  transformers 4.57, etc.). The production 0.17 tool was left
  untouched.
- `brew install llama.cpp` → `llama-server` **b9780** (the exact path
  Surya's own error hint recommends).
- First run auto-downloaded the model: `datalab-to/surya-ocr-2-gguf`
  (`surya-2.gguf`, ~1.4 GB) + an mmproj (vision projector) to
  `~/.cache/huggingface`. `SuryaInferenceManager` auto-spawns
  `llama-server`.
- Test page: Frantz Fanon, *A Dying Colonialism*, p.25, body text,
  rasterized to 1100×1600 (a 4887×7112 200-dpi render was also tried).

## Result: does not work out of the box

OCR returned **empty text** and layout returned a single bogus
`PageHeader` region on every configuration tried:

| Config | layout | ocr | notes |
|---|---|---|---|
| guided on (default) | 0 regions, 128s | empty | `llama-server` 400: "Failed to initialize samplers: **failed to parse grammar**" |
| `SURYA_GUIDED_LAYOUT=false` | 1×PageHeader | empty | no grammar error, but degenerate |
| + `SURYA_INFERENCE_PARALLEL=1` | 1×PageHeader | empty | same |
| + `LLAMA_CPP_EXTRA_ARGS=--image-min-tokens 1024` | 1×PageHeader | empty | same |

## Root cause (from `~/.cache/datalab/surya/llamacpp_server.log`)

The model **is** Qwen-VL and the image **is** being encoded
(`process_mtmd … n_chunks = 1`, ~1860-token prompt). But generation
stops after **13–15 output tokens** amid:

- `find_slot: non-consecutive token position 7 after 7 …`
- `forcing full prompt re-processing due to lack of cache data
  (likely due to SWA or hybrid/recurrent memory)` —
  [llama.cpp PR #13194](https://github.com/ggml-org/llama.cpp/pull/13194)
- `load_hparams: Qwen-VL models require at minimum 1024 image tokens…`

So this is a **backend incompatibility between the Surya 2 GGUF and the
current Homebrew llama.cpp build** — the model's sliding-window
attention / KV-cache checkpointing trips a llama-server bug that
truncates output to near-nothing. The grammar-parse 400 (guided
decoding) is a second, independent incompatibility. Neither is a
problem in our code — they happen with a bare standalone script using
Surya's own API + recommended install.

## Verdict: NO-GO for now (revisit when the backend stabilizes)

1. **Runtime is not production-ready on Apple Silicon with stock
   tooling.** The documented `brew install llama.cpp` produces a
   `llama-server` that can't run the Surya 2 model correctly. Making it
   work would mean pinning a specific (patched/newer) llama.cpp build —
   a moving target tied to upstream SWA/grammar fixes.
2. **The deployment burden is much heavier than today.** Surya 1 ships
   to users via one `uv tool install` and runs in-process on MPS.
   Surya 2 adds: Homebrew, a specific llama.cpp build, an auto-spawned
   server process (port/lifecycle/log management), model + mmproj GGUF
   downloads, and server-flag tuning. That's a large `SuryaSetupSheet`
   and `SuryaConnection` rewrite for a non-technical audience.
3. **Even the API contracts shift** (already mapped, for when we do
   migrate): OCR `text_lines[].text` → `blocks[].html` (block-level,
   not line-level — disturbs paragraph-bbox alignment); table cells
   lose `is_header/rowspan/colspan` but `TableRecPredictor.predict_full`
   returns `<table>` HTML directly (a cleaner path);
   `DetectionPredictor` stays torch-only.

**Recommendation:** stay on the pinned Surya 1 (`>=0.17,<0.20`) shipped
in Phase 0. Re-spike when (a) a Homebrew/llama.cpp release runs the
Surya 2 GGUF cleanly end-to-end, or (b) datalab ships a self-contained
macOS runtime. Keep the `sidecar_v2.py` exploration and the contract
mapping above for that revisit.

## Reusable assets

- Standalone spike script: `scratchpad/spike_v2.py` (no Humanist deps;
  layout + OCR + optional table HTML, with timing + peak RSS).
- The HF model cache (`datalab-to/surya-ocr-2-gguf`, ~1.4 GB) and the
  isolated venv remain available for a quick re-test against a future
  llama.cpp build.
